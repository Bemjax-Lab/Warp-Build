PWA setup — what makes it installable

PieceWhat it does
manifest.webmanifestApp name, short name, icons (192² + 512²), theme colour, display: standalone, start URL. Without this the browser will not show "Add to Home Screen".
Service workerCaches the app shell so it loads offline. We make the parameters + localisation flow work without network; case sync happens when connection returns.
Apple touch iconsiOS does not honour the manifest icon for "Add to Home Screen". Add <link rel="apple-touch-icon"> for 152×152 and 180×180.
Splash screens (iOS)iOS expects per-device-resolution splash images. Generate from the icon + brand colour.
Maskable iconAndroid Adaptive Icons — the icon survives the device's mask cropping. Generate alongside the standard icon.
HTTPSPWAs only install over HTTPS. Vercel / Cloudflare give this for free.

"Add to Home Screen" — test on real devices

  • iOS Safari: Share menu → Add to Home Screen. Test on an iPhone running current iOS and one a couple of versions back. iOS does not show the install prompt automatically — we need a small in-app banner explaining the path.
  • Android Chrome: Browser shows an "Install app" prompt automatically when the manifest is valid and the user has engagement signals. We can also trigger the prompt from a button (BeforeInstallPromptEvent).
  • Desktop Chrome / Edge: Install icon appears in the address bar. Useful for in-clinic desktops.

Stripe — payments on the web

StepDetail
Stripe accountBusiness info, bank account in entity's country, tax info. Country choice tied to the legal entity.
Subscription productsMonthly + annual + early-bird tier. BRL pricing for BR (Stripe supports local card networks).
Checkout flowStripe Checkout (hosted page) is the simplest. Stripe Elements for an inline custom flow if Checkout cannot be styled to brand.
Customer portalStripe-hosted page where users update payment method, cancel, view invoices. Saves us building one.
Webhookscustomer.subscription.* and invoice.payment_* events update our database. Idempotent by event ID.
Free trial handlingTrial without card upfront → user signs up, has 14 days, sees "add card to continue" on day 14. Higher activation, lower trial-to-paid conversion.
Tax handlingStripe Tax (1% fee) computes VAT / sales tax / Brazilian taxes per region — saves us building a tax matrix.

Web-specific operational notes

  • No review queue means we can ship multiple times a day. Automated tests gate deploys (cypress / playwright on the critical flow: signup → trial → checkout → engine result).
  • The PWA is served from the same origin as the Stripe Checkout return URL — required for the Stripe customer object to bind cleanly to our user account.
  • Service-worker caching needs a versioning strategy (cache name includes a build hash) — otherwise users pin an old version of the app and miss bug fixes.
  • iOS Safari has historically reset PWA storage after 7 days of inactivity. We account for this by re-fetching from the backend on launch rather than depending on local storage as the source of truth.

Order of operations — web

  1. Domains registered (.com + .com.br).
  2. Hosting + database provisioned. TLS certificate issued automatically.
  3. Authentication wired (email + password / magic link).
  4. Stripe account approved → subscription products configured → webhooks pointed at our backend.
  5. End-to-end test on a real device: signup → trial → charge → cancel → reactivate.
  6. PWA manifest + service worker shipped → "Add to Home Screen" tested on iOS Safari and Android Chrome.
  7. Privacy Policy + Terms live (see legal.html).
  8. Soft launch — share the link with the early-bird cohort. No review, no waiting.
